# Security Patterns — playwright-ai-triage

> Generated by agentic-os guide-generator (0.3.0).
> owner: generated — regenerate via /agentic-upgrade; edits survive as owner: user.

Repo-wide secret handling and agent conduct rules live in
[`safety-policy.md`](../policy/safety-policy.md) and are not restated here.
This guide covers the security properties of the code itself.

## A. The trust boundary is data egress — two destinations only

Test-failure text leaves the user's machine to exactly two kinds of
destinations, both user-controlled: the Anthropic API (classification) and the
outputs the user enables (stdout / their GitHub PR / their Slack webhook).
Nothing else. No telemetry, no third-party endpoints, no hosted platform.
Evidence: `README.md` — "Failure text is sent to two kinds of destinations, both under your control: the Anthropic API … and the outputs you enable".

**Rule: never add a new outbound destination without human escalation** — it
changes the product's privacy contract, and `escalate_on` in
[`escalation-policy.md`](../policy/escalation-policy.md) treats it as a
security flag.

## B. Every payload field declares its truncation and redaction limits

`FailurePayload` is the allowlist of what the classifier may see, and each
field's doc comment in `src/types.ts` pins its limit: `errorMessage` and
`stack` truncated to 2000 chars (stack additionally strips `node_modules`
frames), `domSnippet` truncated to 1500 chars and redacted, `diffSummary`
truncated to 1000 chars. A new payload field ships with its limit and
redaction behavior stated in the same doc-comment style, and `README.md`
§"What data is sent where" is updated in the same change.
Evidence: `src/types.ts` — `FailurePayload` doc: "Everything the classifier is allowed to see about one failed test."; per-field comments `/** truncated to 2000 chars */`, `/** … truncated to 1500 chars, redacted */`.
Evidence: `README.md` §What data is sent where — mirrors the same limits.

**Never sent anywhere:** screenshots, videos, traces, or source code beyond
the truncated stack frames — media files are referenced by local path in the
summary, never uploaded (`README.md` §What data is sent where).

## C. Extra data collection is opt-in, default off

Anything that widens what leaves the machine defaults to off and requires an
explicit option: `includeDom` defaults to `false`; the diff summary comes only
from an explicit `GIT_DIFF_SUMMARY` env or a computed `git diff --stat HEAD~1`.
New collection surfaces follow the same opt-in default.
Evidence: `src/types.ts` — `includeDom` doc: "send a redacted DOM snippet with each failure; default false"; `diffSummary` doc: "opt-in via GIT_DIFF_SUMMARY env or computed `git diff --stat HEAD~1`".

## D. Credentials come from env only, never from options or code

`ANTHROPIC_API_KEY`, `GITHUB_TOKEN`, and `SLACK_WEBHOOK_URL` are read from the
environment; the `AiTriageOptions` surface carries no secret-typed field, and
none may be added. Tests never use real keys — they mock the Anthropic SDK
(`CONTRIBUTING.md`: "Never commit secrets or real API keys; tests mock the
Anthropic SDK"). Secret-file deny patterns are pinned in
[`safety-policy.md`](../policy/safety-policy.md) §Secret-bearing files.
Evidence: `src/types.ts` — `AiTriageOptions` doc names the three env vars; no option field holds a credential.

## E. Untrusted model output is schema-validated, failure degrades safely

Classifier output is untrusted input: it is validated against the
`Classification` schema, and anything that fails validation becomes
`UNCLASSIFIED` — never a crash, never blindly-trusted text. The same
degradation ladder applies to the whole reporter: no API key ⇒ plain failure
summary; API down ⇒ `UNCLASSIFIED`; any internal error ⇒ logged warning, run
exits normally. The reporter must never throw from a hook or affect the host
build's exit status (tested — see
[`development-practices.md`](development-practices.md) §D).
Evidence: `src/types.ts` — `Classification` doc: "Schema-validated classifier output; anything that fails validation becomes UNCLASSIFIED."; `FailureClass` includes `'UNCLASSIFIED'`.
Evidence: `README.md` — "The reporter never fails your build. No API key? It degrades to a plain failure summary."
weak-evidence: the zod validation itself is not yet implemented (M0 skeleton; `zod` is a dependency in `package.json` but unused in `src/`) — the rule is the pinned design contract to implement against, confirm once classification lands.

## F. Supply chain: minimal publish surface, minimal CI permissions, provenance

The published package contains only `dist`, `README.md`, `LICENSE`, and
`CHANGELOG.md` (`package.json` `files` array) — never add config or local
paths to it. CI runs with `permissions: contents: read`; publishing happens
only in `release.yml` with npm provenance enabled (`NPM_CONFIG_PROVENANCE:
'true'`) and is human-gated via the merge flow. New runtime dependencies
beyond `@anthropic-ai/sdk` + `zod` are a design decision — escalate first.
Evidence: `package.json` — `"files": ["dist", "README.md", "LICENSE", "CHANGELOG.md"]`; two-entry `dependencies` block.
Evidence: `.github/workflows/ci.yml` — `permissions: contents: read`.
Evidence: `.github/workflows/release.yml` — `NPM_CONFIG_PROVENANCE: 'true'` on the publish step.
