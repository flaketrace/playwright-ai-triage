## A. System shape: a single-entry npm library, nothing to run

playwright-ai-triage is a Playwright test reporter distributed as an npm
package. There is no server, no database, no UI, and no start command — the
package is loaded by the host project's `playwright.config.ts` (`reporter:
[['list'], ['playwright-ai-triage']]`) and runs inside the host's test
process. The whole runtime is one class plus its types.
Evidence: `src/index.ts` — `export default class AiTriageReporter implements Reporter` is the entire runtime entry.
Evidence: `package.json` — no `dev`/`start` script; `@playwright/test` is a `peerDependency` (`>=1.40.0`), supplied by the host project, never bundled.
Evidence: `README.md` §Planned usage — the one-line config integration.

## B. Data flow (the design contract the types pin down)

The current implementation is deliberately an inert M0 skeleton — hooks are
wired but empty (`src/index.ts` docstring: "hooks are wired but inert;
collection, classification, and rendering land in the next releases"). The
flow the type layer already pins:

1. **Collect** — Playwright hooks (`onBegin` / `onTestEnd` / `onEnd`) gather
   failures into `FailurePayload` objects (truncated/redacted per
   [`security-patterns.md`](../development/security-patterns.md) §B), capped
   at `maxFailures` (default 25).
2. **Pre-pass** — deterministic heuristics compute `retryThenPassed` and an
   optional `heuristicPrior` ("passed to the model as evidence, not a
   verdict" — `src/types.ts`).
3. **Classify** — the Anthropic API returns a `Classification`
   (schema-validated; invalid ⇒ `UNCLASSIFIED`). `dryRun: true` substitutes
   fixture classifications with no API call.
4. **Render** — a summary posts to the enabled `outputs` (`stdout` always;
   `github` auto-detected from GitHub Actions env; `slack` when
   `SLACK_WEBHOOK_URL` is set).

Evidence: `src/types.ts` — `FailurePayload`, `Classification`, `FailureClass`, and the `outputs`/`maxFailures`/`dryRun` option docs encode each stage.
Evidence: `src/index.ts` — the three wired hooks (`onBegin`, `onTestEnd`, async `onEnd`).

## D. Distribution: dual ESM/CJS from one source tree

tsup builds `src/index.ts` into ESM (`dist/index.js`), CJS (`dist/index.cjs`),
and type declarations, targeting `node18` to match the `engines` floor
(`>=18`). The `exports` map in `package.json` routes `import`/`require`/
`types` to the matching artifact. `dist/` is generated and gitignored — source
changes only ever happen in `src/`.
Evidence: `tsup.config.ts` — `format: ['esm', 'cjs']`, `dts: true`, `target: 'node18'`.
Evidence: `package.json` — `exports['.']` with `types`/`import`/`require` conditions; `"engines": { "node": ">=18" }`.

## E. Known architectural limits (do not silently "fix")

Two limitations are documented as v1 scope decisions, not bugs: sharded runs
(`--shard`) post one summary section per shard with no cross-shard merging,
and the PR-comment output is GitHub-only (`stdout`/`slack` work on any CI).
Changing either is a scope expansion — escalate, don't drive-by implement.
Evidence: `README.md` §Known limitations — both items stated verbatim.
